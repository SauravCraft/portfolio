PROJECT // 001
Networked Inventory System
Problem
Standard UE5 array replication caused full-array resends on any mutation — causing latency spikes during concurrent 4-player inventory operations, unacceptable for competitive multiplayer.
Architecture
Server-authoritative storage using a custom FFastArraySerializer subclass with per-item dirty flags. All mutations (add, remove, move, use) are routed through validated server RPCs. Client UI updates via OnRep delegate broadcasts.
Implementation
Delta Replication ensures only changed item slots are sent per tick. Weight limits, slot capacity, and item stacking logic are all enforced server-side with client-predicted visual feedback to maintain responsiveness.
Result
Dramatically reduced per-operation bandwidth. Supports 4-player concurrent inventory mutations with sub-1ms delta sync. Zero desync errors during playtesting sessions.

PROJECT // 002
Dungeon Escape
Problem
Non-linear dungeon traversal required dynamic room loading without stalling the game thread or causing memory overflow from holding all rooms simultaneously.
Architecture
Modular sub-level streaming system — each dungeon room is an independently loaded/unloaded sub-level. Proximity trigger volumes call async load/unload, keeping only 2–3 adjacent rooms in memory at any time.
Result
5 unique dungeon layouts with zero hitches during traversal. Memory budget held under 128MB with async streaming preventing any game-thread stalls.

PROJECT // 003
Parkour Game
Problem
Complex traversal movement (wall-running, vaulting, ledge grabbing) needed to feel responsive and physically grounded — any input lag or jank breaks player immersion completely.
Architecture
FSM-driven movement extending UCharacterMovementComponent. Wall detection via multi-trace capsule sweeps. Ledge detection uses async physics line queries with IK foot placement. All states managed by a single MovementFSM component.
Result
Sub-frame input responsiveness across all 6 movement states. Wall-running, vaulting, ledge grabs, air control, and sliding all transition correctly without state conflicts.

Battle Blaster
Problem
An arcade shooter needs fast weapon switching, balanced enemy spawning, and satisfying projectile physics — all while keeping combat feel tight and responsive under heavy AI load.
Architecture
Abstract AWeaponBase class with Rifle, Shotgun, and Rocket subclasses. Enemy spawning uses an ASpawnPool actor with wave-based escalation. All damage is routed through a generic IDamageInterface for full decoupling.
Result
Playable arena level with 8+ simultaneous AI enemies, seamless weapon swap, and Niagara particle-driven combat feedback. Wave progression tested to Phase 5 without frame drops.
